# Xへの投稿作成ワークフロー (Claude Code用)

このワークフローは、Claude CodeがX（旧Twitter）への投稿文を作成・提案・登録する際の一連の手順を定義するものです。

## 1. 情報収集・コンテキスト理解（調査フェーズ）
- 元となる情報源（Notionの記事、ニュース、ユーザーからの提供テキストなど）を読み込む。
- ターゲットオーディエンスと、投稿するアカウントのペルソナ・トンマナ（例：「ふくろう（@hukuro_educator）」などの専門性・語り口）を確認する。
  - **重要**: アカウントのペルソナを設定する際、安易な語尾（例：「〜ホ」など）の付与は避けること。Notionのナレッジベース、会話ログ、日報などからユーザーの実際の価値観や行動、本来のトーンを読み取り、事実ベースで説得力のある投稿を作成する。
- 既存のX投稿内容DBを確認し、切り口を変えるなどして内容の重複を避ける。
- 訴求すべきコアなメッセージ（事実・課題・解決策・スタンスなど）を1〜2行で定義する。

## 2. プランニング・構成案の作成（なまいきスクール式）
- **逆算意識の徹底**: Xマネタイズの4ステップ（認知→興味→リスト化→セールス）を意識し、今回のポストが「認知獲得」「興味付け」「教育」などのどの目的を担うか必ず言語化する。
- **ターゲット設定**: 知識ゼロの「0〜1段目の人」（フォロワーの90%）に向けて、中学生でも一発で理解できる具体度と言葉選びを徹底する。
- **フェーズに合わせたムーブ**: 
  - 発信初期（インプ1000未満）は「教えてあげる」等の先生ムーブは絶対に避け、「プレイヤー（活動している当事者）」としてちょい自慢風や行動実況のポストを行う。
- アカウントの目的に合わせて、どのような切り口（型）で投稿するか方針を立てる。（例：専門性重視の解説スタイル、共感重視の煽りスタイル、図解ベースなど）
  - 日曜系（興味づけ）と有益ポストの比率は指示がない限り7:3程度を意識し、興味づけを多めにする。興味付けは「実績」だけでなく「目新しさ・再現性・即効性・常識破壊・損失回避」のワードを使う。
  - 画像ありの投稿を多めに想定し、必要なスクリーンショットのイメージを明確（例：「NotionのXXX画面のスクショ」）にしてユーザーに依頼する。
- 必要に応じて、構成要素（Hook、Context、Main point、Takeaway、CTA）を整理する。特に**ノンバーバル教育（背中で見せつけるストーリー型：宣言→途中経過→ピンチ/チャンス→オチ）**を意識する。

## 3. ポスト文のドラフト作成（実行フェーズ）
- `content-creator` スキルのガイドライン（Twitter/X Threadsの型など）を参照し、文字数や可読性に配慮してドラフトを作成する。
- **なまいきスクール式 投稿テクニック**:
  - **冒頭1文目が命**: ターゲットと課題を明示し「誰向けか」をN=1に刺す。（例：❌「Xをはじめて3ヶ月…」→⭕「Xを3ヶ月続けても伸びない人が100%やっていないこと」）
  - **インセンティブ設計**: CTA（フォロー、リプ、通知オン、LINE登録）には必ず「読者のメリット」を提示する。単なる「お願い」はしない。
  - 長文構成: 「結論から書かない → 感情を揺らして焦らす → 価値提供 → アクション」の構成で滞在時間を伸ばす。
  - **リスト・構造化の活用**: 情報量が多い長文ポストでは、視認性を高めるために箇条書き（・）や番号付きリスト（1. 2. 3.）を積極的に用いる。
- **厳守事項（フォーマット）**:
  - URLリンクは一切含めないこと。
  - 出力テキスト内に `\n` という文字列（改行コードの文字列表現）を混入させないよう注意すること。
  - 引用リポスト（Quote RT）の形式は使用しないこと（精度担保のため）。
- 一つの事象に対して、異なる切り口やトーンを用いた**3〜4パターンのバリエーション**を提示し、ユーザーに選択肢を与える（複数本を一度に作成する場合はこの限りではない）。

## 4. レビューと修正（ユーザー確認）
- 作成したドラフトをユーザーに提示し、フィードバックを求める（`notify_user`）。
- 指摘があれば、それに従ってトーンや内容を微調整し、再提案する。
- 修正から得られた気付きは `tasks/lessons.md` に追記して次のタスクで同じミスを防ぐ。

## 5. 投稿DBへの登録（完了フェーズ）
- ユーザーから最終承認を得たポストを、指定されたNotionの自動投稿用データベース（例：Sparkie Posts）、または該当する管理ファイルに API / ツール等 を用いて登録する。
- 画像が必要な場合はルール（独自図解/スクショ等の使用など）に従って処理し、Notionブロック等に正しくアタッチするか、プレースホルダーとして指示を残す。
- 完了をユーザーに報告する。

---
**システムプロンプト/セルフリマインダー**:
Xの投稿作成を依頼されたら、すぐにテキストを生成し始めるのではなく、**まずプランニングと方針策定（ステップ1＆2）をユーザーに見せて合意をとること**。構成が決まってからドラフト作成（ステップ3）に進む。

### 投稿データ作成時の特別ルール
- **タイトル設定**: Notionの「タイトル（1文目フック）」プロパティには、投稿文の **1文目** をそのまま設定すること。
- **本文データ**: 投稿文の全テキストは、ページ内の **本文（子ブロック）** に直接書き込み、必ず**コールアウトブロック**（💡アイコンなどを指定）で囲むこと。
- **準備完了フラグ**: 自動投稿を予期せずトリガーしないよう、「準備完了」プロパティはユーザーの最終確認が終わるまで **未チェック(false)** にしておくこと。
- **プロパティの網羅**: 「🚫」のマークがついているプロパティ以外は、基本的にすべて埋めること。常にポストの目的や全体の文脈を強く意識し、それらに沿った適切な値（タグ、状態、意図など）を設定する。
