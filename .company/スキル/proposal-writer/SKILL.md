# Proposal Writer — クラウドソーシング応募文作成スキル

**営業部門・案件対応担当が応募文を作成する際の必須参照スキル。**
オーナー上原がそのままクラウドワークス・ランサーズ・ココナラのフォームに貼り付けて応募できる完成形ドラフトを作る。

---

## 大原則

1. **「普通すぎる応募文」はNG**。テンプレで埋めず、案件固有の洞察を1段落以上必ず入れる
2. **Notionへの書き込みは決まったフォーマット**で行う（後述）
3. **選考中の応募文と同じ密度・同じ熱量・同じ文体**で書く。短くまとめない
4. **オーナー上原が読み返したときに「自分が書いた」と思える文章**にする
5. **募集本文を読み込み、案件特有の語彙・論点を必ず2〜3個拾って文中に織り込む**

---

## Notion書き込みフォーマット（厳守）

案件ページの本文を `replace_content` で以下の形に置き換える。

````
## 応募文下書き
```javascript
（応募文本文をここに書く。改行・空行は実際の改行で表現する）
```
````

ポイント:
- 見出しは `## 応募文下書き`（H2）。`# 応募文下書き`（H1）ではない
- コードブロックの言語は `javascript`（Notion上での視認性が良いため統一）
- 応募メモ・補足はコードブロックの**外**に書いてよいが、応募文本文は必ずコードブロック内
- 書き込み後は必ず `notion-fetch` で1件再取得し、改行・URL・装飾が崩れていないか確認する

## 構成テンプレート（必須7ブロック）

選考中応募文の共通構造。すべての応募文はこの順で書く。

### ① 自己紹介（1〜2行）
```
はじめまして。
〇〇（案件カテゴリに合わせた肩書）を行っております、上原と申します。
```
- 肩書はカテゴリ別表（後述）から選ぶ
- 「上原 千翔」とフルネームにしない（締めで使う）
- 「フリーランスで」を頭に付けてもよい

### ② 案件固有の洞察ブロック（2〜4行）★ここが「普通じゃない」要
募集本文から拾ったキーワードを使って、ターゲット心理・業界課題・LP/プロダクト特有の論点を**自分の言葉で言語化**する。テンプレ流用は即NG。

例（RilenScotch オンラインサロンLP）:
> 20代独身社会人の方が「ちょっと覗いてみようかな」と思えるような、気軽さと信頼感を両立したページが求められるかと思います。
> オンラインサロンのLPは、コミュニティの雰囲気が伝わるかどうかが申込みへの分岐点になりますので、ファーストビューで空気感を伝え、スクロールの中で「自分にも合いそうだ」と感じてもらえる流れを意識して制作いたします。

例（media_works AI×BtoB LP）:
> BtoB向けのLPでは、ターゲットである企業の役員やマーケティング担当の方が「自社にも関係がある」と感じられるファーストビューが成否を分けます。
> テキスト設計書の訴求構造を活かしつつ、信頼感と専門性が伝わるデザインをご提案いたします。

書き方のコツ:
- 「〇〇な方が『△△』と思えるような」型でターゲット心理を言語化
- 「〇〇のLPは、△△が分岐点になりますので、〜を意識して制作いたします」型で論点を1つ提示
- LP案件なら ファーストビュー / CTA / 離脱対策 / 視線誘導 / トンマナ / コミュニティの空気感 などの語彙を1〜2個入れる
- AI/Notion案件なら 属人化 / 業務フロー / 自走化 / リレーション設計 / 現場定着 などを使う

### ③ ■デザイン面の信頼担保（LP案件のみ・必須）
```
■ デザイン面の信頼担保

コーディングだけでなく、デザインの土台となる経験・知識も持ち合わせております。
・イラストレーターとしての制作経験
・チラシデザインの制作経験
・色彩理論・デザインの基礎知識

「ただ作る」だけでなく、ターゲット心理に響く配色・余白・視線誘導まで設計できます。
```
LP案件・LPコーディング案件・Webデザイン案件には**必ず差し込む**。技術力だけだとエンジニア未経験で弱いので、デザイン側の素養で信頼を担保する。

### ④ ■提案内容（進め方）
番号付きステップで3〜5項目。案件のスコープに合わせて段階を設計する。

LP案件の標準フロー:
```
■ 提案内容
1. ヒアリング（オンライン）
   ・サービスの強み・ターゲット・競合との差別化ポイントを整理
   ・LPのゴール（問い合わせ／資料請求／申込み等）を明確化
2. 企画構成・ライティング
3. デザイン制作（スマホファースト）
4. コーディング・レスポンシブ対応
5. フィードバック・修正・納品

AIを活用した開発体制により、低価格帯でも高品質なLPを提供しております。要件が固まっていれば3日以内での納品も可能です。
```
段階発注の案件なら、その段階に合わせて再構成する。

### ⑤ ■経歴（3〜5行）
- 学習塾講師＋フリーランスの組み合わせを軸に
- 「複雑な内容をわかりやすく構造化して伝える」「ターゲット心理に寄り添った構成設計」を強調
- 案件カテゴリに刺さる経歴のみ厳選

例:
```
■ 経歴
学習塾で講師をしており、複雑な内容をわかりやすく構造化して伝えることを日常的に行っています。
LP制作とNotion業務基盤設計を主軸に活動しており、ビジネスサービスの訴求設計にもその経験を活かしております。
```

### ⑥ ■実績（LP実績リスト or 自社プロダクト紹介）
LP案件は4件のLPリンクを基本セットで貼る:
```
■ 実績
企画構成からコーディングまで一貫して制作したLPの実績です。

・自社プロダクトNotion BizBase LP：https://kazutoue45-glitch.github.io/notion-bizbase-portfolio/
・探求プレゼンバトルLP：https://kazutoue45-glitch.github.io/tankyu-presen-battle-lp/
・ブランド物販LP：https://kazutoue45-glitch.github.io/brand-butsuryu-lp/
・カルマ垢LP：https://kazutoue45-glitch.github.io/karma-lp/

スマホファースト設計を徹底しており、レスポンシブ対応も含めて納品いたします。
```

Notion構築・AI案件は Notion BizBase の紹介URLを軸に。

### ⑦ 締め
```
ご検討のほどよろしくお願いいたします。
```
- 「ご判断はお任せ」「お力になれましたら幸いです」等の冗長社交辞令はNG
- 募集要項に「面談前提」と書かれていれば「ぜひ一度お話しできればと思います」を1行追加してもよい

## 参考文3（選考中 — AI×業務改善 / クラウドワークス・バドレックス1111）

ストーリー型応募文のお手本。経験を「どう考えて、何に迷って、どう動いたか」で語る。

```
はじめまして。
NotionでAI業務基盤設計を行っている上原と申します。

「ツールに詳しい人ではなく、業務課題に向き合った経験のある人」という募集文を読んで、まさに自分の仕事のスタンスと重なると感じて応募しました。

以下、ご指定の7項目に沿って、過去の案件を共有します。

① 案件の概要
個人経営の学習塾（講師5名・生徒約80名）。紙とExcelで運営していた業務全体を、Notionベースの業務基盤に移行するプロジェクトです。

② 企業側が抱えていた課題
生徒情報・講師シフト・売上管理・保護者連絡がすべてバラバラのツールに散らばっていて、「誰が何をどこまでやったか」が把握できない状態でした。月末の請求作業に毎回丸1日かかっていたのが一番の痛みでした。

③ 行った対応と、そのアプローチを選んだ理由
Notionに生徒DB・講師DB・授業記録DB・売上DBを作り、リレーションで連携させました。あえてNotionを選んだのは、現場の講師がITに強くなかったから。「見た目でわかる」「触って壊れない」ツールでないと定着しないと判断しました。
加えて、Claude Codeを活用した自動処理（月次レポート生成・リマインド通知）も組み込みました。

④ 一番難しかったこと
設計よりも「現場に使ってもらうこと」が圧倒的に難しかったです。最初は講師の方々が従来のExcelに戻ろうとする場面が何度もありました。週1で15分の使い方レクチャーを3ヶ月続けて、ようやく定着しました。

⑤ 成果や変化
月末の請求作業が丸1日→約1時間に短縮。講師間の情報共有がリアルタイムになり、生徒の状況把握のための会議が不要になりました。

⑥ 使用したツール・技術
Notion（DB設計・リレーション・ビュー設計）、Claude Code（AIエージェントによる自動処理）、Google Apps Script（外部連携）

⑦ 今振り返ると「こうすればよかった」と思う点
初期段階でもっとヒアリングに時間をかけるべきでした。業務フローを最初にしっかり可視化してから設計に入るべきだったと思います。

ぜひ一度お話しできればと思います。
ご検討のほどよろしくお願いいたします。
```

ストーリー型のポイント:
- 数字を必ず入れる（80名・5名・丸1日→1時間）
- 「あえて〇〇を選んだのは△△だから」型で意思決定を語る
- 「設計より現場に使ってもらうことが難しかった」型で失敗・苦労を正直に書く
- 反省を「もっと〇〇すべきだった」型で書く

## 文体ルール（厳守）

### ✅ 必須
- **「ですます」敬語ベース**で統一
- 改行・空行を多めに使い、視覚的に読みやすく
- ■見出しの前後に空行を入れる
- 番号付きリストは半角数字＋ピリオド、補足は2スペースインデント＋「・」
- 「〜とのことで、〜と考えております」型で所感を表明
- 「〜かと思います」型で控えめに仮説を出す
- 「〜を意識して制作いたします」「〜まで設計できます」型で能動表現

### ❌ 禁止
- **「！」「？」「…」を使わない**
- **絵文字・顔文字なし**
- **「ぼく」NG** → 「私」or 一人称省略
- **「めちゃくちゃ」「正直」「ぶっちゃけ」等のカジュアル語NG**
- **「いかがでしたでしょうか」「まとめると」等のAI定型句NG**
- **「なぜなら」「しかし」「つまり」「一方で」等の教科書接続詞NG**
- **括弧（）を使った補足説明を避ける**。同段落で2回以上はNG。読点や → で代替
- **「Notion BizBase」を略さない**。「ビズベース」「BizBase」単独・「Pro」付きはNG
- **「元教育者」と書かない**。現役の学習塾講師
- **「本業として取り組みます」と書かない**。複数案件並行スタイルなので一案件専念は約束しない
- **「必ず期待以上の〜」等の定型句を全案件で使い回さない**
- **「ご判断はお任せ」「お力になれましたら幸いです」等のノイズ社交辞令禁止**
- **相手が知っている情報を説明しない**（上から目線になる）
- **経験がないことを「対応可能」と書かない**（WordPress経験なし→「対応可能」はNG）
- **PSD納品対応と書かない**
- **Figmaをメインツールとして書かない**。コードファースト（Claude Code → GitHub）が主軸
- **都内出社必須の案件には応募しない**（オーナーは大阪在住）
- **実績素材にない経歴・スキルを捏造しない**
- **4件以上の応募文で同じ構成・同じ表現を使い回さない**（②の案件固有洞察ブロックを毎回書き分けることで回避）

## ワークフロー

1. **募集本文を読み込む**
   - Notionの案件ページから取得
   - URLが取得できれば WebFetch で原文も取得（必須項目漏れを防ぐ）
2. **相手の課題・ニーズを3行で整理する**（メモ）
3. **案件固有の洞察ブロック②を先に書く**。ここがテンプレ感を消すキモ
4. **構成テンプレ①〜⑦に沿って本文を組み立てる**
5. **募集本文に必須質問があれば、⑥と⑦の間に必須項目回答セクションを追加**
6. **禁止事項チェック**を全項目走らせる
7. **案件ページに `## 応募文下書き` ＋ ` ```javascript ` コードブロックで書き込む**
8. **書き込み後、必ず `notion-fetch` で再取得して改行・URL・装飾を確認**
9. **崩れていれば即修正**
