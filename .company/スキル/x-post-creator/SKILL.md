## 1. 絶対禁止事項 — AIっぽさを殺すための縛り

**最重要セクション。** このルールを1つでも破った時点で、その投稿は不合格。

### 1-A. 文体の禁止事項

| 禁止パターン | 理由 | 代替表現の例 |
|---|---|---|
| 冒頭や結びの問いかけ：「〜していませんか?」「〜ですか?」 | AI量産の典型パターン | 感情の爆発・事実の叩きつけで始める |
| 綺麗ごと・教訓まとめ：「〜が大切です」「〜を心がけましょう」「〜が重要です」 | 温度ゼロの道徳 | 自分の行動で断言するか、事実で余韻を残す |
| 教科書的な接続詞：「なぜなら」「しかし」「つまり」「一方で」 | 論文臭い | 感情の流れや事実の羅列で繋ぐ |
| 過剰な丁寧さ・謙遜：「〜だと思います」「〜かもしれません」 | 弱腰で刺さらない | 「〜だ。」「〜していく。」「〜でしかない。」 |
| 上から目線・説教：「〜すべき」「〜の時代は終わりました」「〜を見直しましょう」 | 先生ムーブ・偉そうに聞こえる | 自分の体験として語る。「ぼくは〜に変えた」「〜に気づいた」 |
| AI頻出語彙：「構造化」「自律的」「環境構築」「一元管理」「最適化」「シームレス」「ユーザー体験」「パラダイムシフト」「本質的に」「圧倒的に」「革新的」「網羅的」「体系的」「飛躍的に」 | AI系発信者のポストで使い回されすぎて一発でバレる | 動作と数字に置き換える。「構造化した」→「顧客ページ開いたら過去のやり取り全部見えるようにした」 |
| 抽象的なビフォーアフター：「劇的に変わった」「一気に楽になった」「全然違う」 | 数字も具体動作もないので何も伝わらない | 必ず数字か具体動作を入れる。「準備15分→指示5秒になった」「500文字コピペしてたのがワンクリックになった」 |

### 1-B. 記号・文字の禁止事項

- 全角・半角を問わず**括弧類**の一切の使用禁止。
- **ダブルクォーテーション**の使用禁止。
- **リテラル文字列としての改行コード**の出力禁止。実際のLine Feedを使う。
- **「...」三点リーダーの使用禁止**。思考の途切れや余韻を「...」で表現しない。改行、体言止め、「いや、」「てか」等の口語転換で代替する。

### 1-B-2. 絵文字のルール

- **使用可能な絵文字**: ❗️ ‼️ 🔥 😂 😅 😇 😭 😱 🤫 🤔 🦉 — この11種のみ。
- **1投稿あたり最大2種類**まで。同じ種類を複数回使うのはOKだが、3種類以上は禁止。
- 使わなくてもOK。無理に入れない。自然に感情を強調したい箇所にだけ添える。
- 上記リスト以外の絵文字は引き続き**完全禁止**。

### 1-D. 構造のAI臭さを殺す — 最重要追加ルール

AIが書いた文章は「構成が綺麗すぎる」ことで一発でバレる。以下を徹底すること。

| 禁止パターン | 理由 | 対策 |
|---|---|---|
| 毎回同じ4段構成で書く | 予測可能＝読み飛ばされる | 5本中3本以上が同じ構成はNG。構成のバリエーションを出す |
| 全段落が論理的に正しい順番で並ぶ | 人間はそんなに整理して書かない | 思考の飛躍・省略・脱線を意図的に入れる。説明しすぎない |
| 過去形の羅列: 〜した。〜だった。〜できた。 | 作業報告書みたいで温度ゼロ | 体言止め・倒置・口語・現在形を混在させてリズムを壊す |
| フックと結論が綺麗に対応してる | 伏線回収が完璧すぎるとAI臭い | オチは斜めから来るか、回収しきらないまま終わる |
| 1ポストに情報を詰め込みすぎる | 全部伝えようとするのはAI思考 | 1ポスト1感情 or 1事実。余白と行間で語る |
| 「具体→抽象→教訓」の三段論法 | テンプレ感がえぐい | 具体だけで終わる。教訓を言語化しない。読者に委ねる |

**構成のバリエーション例:**
- 感情だけ吐き出して終わる型
- 事実を3つ並べて余韻で終わる型
- 途中まで語って「...まあいいか」で投げる型
- 1つのシーンだけを切り取るスナップショット型
- 自問自答しながら結論が出ないまま終わる型

**人間っぽさのシグナル:**
- 「...」で思考の途切れを見せる
- 「いや、」「てか」「あ、」で話の方向転換をする
- 1文だけ妙に短い文を混ぜる
- 結論を言い切らず体言止めや「...」で余韻を残す
- 同じポスト内で敬体と常体が混ざってもOK

### 1-D-2. AIが陥りやすいパターン — 絶対に避ける

以下はAIが無意識にやってしまう「一発でバレる」パターン。生成後に必ずチェックする。

| 陥りやすいパターン | 具体例 | 対策 |
|---|---|---|
| **自分語りで始まる** | 「ぼくが最初にやらかしたのは」「数ヶ月前のぼくの〜」「ぼくの固定費を〜」 | **1文目は必ず読者の課題・状況・内心の声から始める。** 「ぼく」は2文目以降。読者が「あ、これ自分のことだ」と思えなければ続きは読まれない。プロフィールクリックにつながるのは「この人の話をもっと聞きたい」と思わせた時だけ |
| 過去形の報告調羅列 | 〜して、〜して、〜した。全部完了した感の連続 | 体言止め・現在形で途切れさせる。1ポストに「した」は2回まで |
| 教訓・格言・断言で締める | 「手を止めなかったから今がある」「ゼロから考えない」「ここが勝負所」「仕組みで解決した」「構造が全てだった」「〜が全てだった」「〜でしかない」 | **行動提案**か**事実のスナップショット**で終わる。「2つだけでいいから試してみて」「今日も朝イチでNotion開いた」「それだけの話」。格言っぽい一般化は全部NG |
| 3本以上が同じ結論テーマで着地する | 「情報の置き場所」「仕組み」「構造」で毎回終わる | バッチ内で結論のキーワードを重複チェックする。3本中2本以上が同じ単語で着地したら不合格 |
| 感情が冒頭だけで本文で消える | 1文目は「マジで焦った」なのに2文目から説明モードに入る | 本文中に最低1箇所、内心の声を挟む。「これ自分でやった方が早くない？って何回も思った」「いや待て、これ違うなって」 |
| 4段構成の固定 | フック→説明→事実→まとめ、が毎回同じ | 7本中5本以上が同じ型は不合格。感情だけ、事実だけ、スナップショットだけ、など型を混ぜる |
| 情報の詰め込み | 1文に複数要素を並べる | 1ポスト1感情 or 1事実。余白と行間で語る |
| 整いすぎた言葉 | 「ユーザー体験は一気に変わる」「ルール1つで迷いが消える」 | 口語・スラングで壊す。「マジで」「えぐい」「てか」を入れる |
| 説明しすぎ | 読者が想像できることを全部書く | 省略する。2割は書かないで余韻に残す |
| 同じ構文の3連打 | 「〜がわからない。〜がわからない。〜が見つからない」「〜はこう、〜はこう、〜はこう」 | 同じ文型を3回以上繰り返さない。2回目で崩すか、別の文型に切り替える |
| 過去形3連打→「でも」転換 | 「〜してた。〜もあった。〜と思ってた。でも〜」 | AIの典型パターン。過去形を連続させず、途中に現在形・体言止め・口語を混ぜる |
| 全投稿が同じ軌道 | 5本全部が「過去の失敗→気づき→改善→今」 | バッチ内で解決しない投稿、事実だけの投稿、堂々巡りの投稿を最低2本混ぜる |

### 1-C. 内容の禁止事項

- **具体的な個人名・アカウント名の記載は絶対禁止**。許可がない限り、固有名詞としての人名は一切出さない。「クライアント」「ある経営者」「とある事業主」など一般名詞に置き換える。引用リポストの場合のみ例外。
- **個人情報の流出禁止**。クライアントの社名、事業内容の特定に繋がる詳細、金額の具体値など、本人の許可なく第三者が特定できる情報は絶対に出さない。
- **ハルシネーション — 事実の捏造は絶対禁止**。やっていないことを「やった」と書く、存在しない実績を語る、DBの一次情報から読み取れない推測を事実として語る — これらはすべて厳禁。切り口を変えることや感情の誇張はOKだが、事実無根の嘘は許されない。
  - **リライト時の特別注意**: 元投稿の事実をベースに切り口を変える。元投稿にない行動・エピソード・数字を追加しない。「サブスクを7つ解約した」「毎日5件質問が来た」のように、日報やDBに裏付けのない具体的行動を捏造するのは典型的なリライト時ハルシネーション。
- **勝手なCTAの挿入禁止**。ユーザーの指示がないのに「フォローして」「リプして」「LINE登録して」等を入れない。
- **同一ネタの繰り返し禁止**。過去に投稿した切り口と同じ内容は避け、常に新しい角度で語る。
- **Difyなど高度な開発ツール**はターゲットが求めていないため、メインにしない。
- **自分のサービスへの直接的な宣伝は1日最大1本**まで。

## 2. 人間味のあるポストの作り方 — 3つの法則

AI感を消し、プレイヤーとしての体温と泥臭さを出すための核心ルール。

### 法則1: フックは「唐突な感情」から入れ

「お悩みの方へ」「〜で困っている人必見」のような親切な前置きは絶対NG。

**今この瞬間の感情や、生々しい状況**をそのまま1文目に叩きつける。

良い例:
- 完全にスイッチ入った。
- 待って意味わからん。
- 深夜やからぶっちゃけるけど
- マジで自分が甘かった。
- えぐいことが起きた。

**重要: フックの感情を本文でも生かす**

1文目で感情を出したら、本文の途中でもう1回その感情に触れる。フックと本文が断絶しないようにする。

- 悪い例: 「マジで焦った。」→ 以降は淡々とした説明
- 良い例: 「マジで焦った。」→ 本文途中で「いやほんとに、画面見て3秒固まった」

フックで出した感情は伏線。本文で1回は回収する。

### 法則2: 「整いすぎた言葉」を壊せ

「非常に重要です」「効率化できます」のような無菌室の言葉は使わない。

**生々しい感情、失敗体験、極端な数字、強めのスラング**を意図的に混ぜて、良い意味で文章の綺麗さを破壊する。

- 「マジでボコボコにされた」
- 「甘さがえぐい」
- 「最低収益75円」
- 「14時間ぶっ通しでやってた」
- 「正直、まだ末端。」

### 法則3: オチは「行動提案」か「事実の断片」で終わる

「いかがでしたか?」「参考になれば幸いです」のような委ね締めは当然NG。
だが、「手を止めなかったから今がある」「ここが勝負所」のような**格言的な断言もNG**。綺麗にまとまるとAI臭い。

終わり方のバリエーション:
- **行動提案型**: 「2つだけでいいから試してみてほしい」「まず日報だけでもNotionに書いてみて」
- **事実断片型**: 「今日も朝6時にNotion開いた。それだけ」「金額計算のロジックを直したくらい」
- **内心の声型**: 「...正直まだわからん」「てかこれ合ってんのかな」
- **途切れ型**: 「続きはまた明日...」「まあ、うん」

禁止する終わり方:
- 「〜が全てだった」「〜でしかない」「〜から今がある」→ 格言。AI指紋。
- 「〜が大事」「〜を忘れるな」→ 教訓。先生ムーブ。
- 「〜という仕組み/構造/環境を作った」→ 抽象的すぎる。具体動作で語れ。

## 3. 企画と構成 — なまいきスクール式

### 逆算意識

Xマネタイズの4ステップを意識し、今回のポストがどの目的を担うか必ず言語化する。

1. **認知** — 存在を知ってもらう
2. **興味** — 中身に興味を持ってもらう
3. **リスト化** — LINE等に誘導する
4. **セールス** — 商品・サービスを購入してもらう

### ターゲット設定

- 知識ゼロの「0〜1段目の人」に向けて書く。フォロワーの90%はここにいる。
- 中学生でも一発で理解できる具体度と言葉選びを徹底する。
- 専門用語を使う場合は正確に。例: CRMは「顧客を中心にした業務基盤」であり、単なる業務管理システムではない。

**0〜1段目の人が「自分ごと」にできるかチェック:**

投稿のネタが高度な場合、以下のどちらかで翻訳すること:
- **入口を下げる**: 「CRM構築した」→「お客さんの連絡先をNotionに全部まとめた」
- **最初の1歩だけ提示する**: 「2つだけでいいから試してみて。ページ1枚作って、名前と連絡先を入れるだけ」

以下は0〜1段目に刺さらない表現:
- DB、リレーション、プロパティ、ロールアップ、フォーミュラ → Notionの機能名をそのまま出すと離脱される
- 「5つのDBを連携させた」→ すごいのはわかるが真似できない。再現性ゼロ
- 「AIエージェントが自律的に動く」→ イメージが湧かない。「寝てる間にAIが日報を書いてくれてた」のように結果で語る

### フェーズに合わせたムーブ

- 発信初期はプレイヤーとしてのちょい自慢風・行動実況のポストが中心。先生ムーブは封印。
- 日常系と有益ポストの比率は**7:3**を意識。日常系を多めにする。

### フックの種類

- 実績・数字
- 目新しさ
- 再現性・即効性
- 常識破壊
- 損失回避
- トレンド
- ストーリー

### ノンバーバル教育

言葉で教えるのではなく、背中で見せつけるストーリー型を意識する。
宣言 → 途中経過 → ピンチ/チャンス → オチ

### インセンティブ設計

CTAには必ず「読者側のメリット」を提示する。単なるお願いはしない。
例: 「フォローして」ではなく「明日のNotion設計の全手順を見逃さないようにフォローしておいて」

### 文脈・ストーリー連続性 — 1日・1週間の投稿設計

何の関係もないただのポストを10連投するより、**文脈がつながってストーリーのあるポスト**の方が反応が上がる。

- **1日・1週間の中で文脈を持たせる** — バラバラのポストの羅列にしない
- **追いたくなる心理** — 続きが気になる構成にすると、過去の投稿も見られやすくなる
- **ストーリーの設計** — 複数本をまとめて作る場合は、テーマ・時系列・体験の流れなどでつなげる

例: 同じテーマの「壁→試行錯誤→突破」を数日に分けて投稿する、週の初めに宣言して週末に結果を出す、など。

**⚠️ 注意: 1文目フックは初見でも成立させる**

文脈でつなげるだけだと、**1文目のフックが弱くなる**危険がある。前の投稿を知らない初見の人が読んだとき「意味わからんこと言ってるな」で離脱されるリスクが高い。

- **各ポストの1文目は、単体でも続きを読みたくなるフックにする**
- 文脈の分割に寄せすぎて、1文目から「ようわからん」にならないこと

## 5. ドラフト作成ルール

### 文字数と形式

- **180文字〜260文字**の間に収める。
- スレッドにせず、**1ポストで完結**させる。
- URLリンクは一切含めない。

### 構成の型 — 4段構成は「1つの選択肢」に過ぎない

**重要**: 「フック→失敗→気づき→まとめ」の4段構成を毎回使うとAI臭くなる。これはあくまで1パターン。7本中2本まで。

**推奨する構成バリエーション:**
- **感情だけ型**: フック→感情の吐露→「...」で終わり。説明しない
- **事実羅列型**: 事実を3つ並べて、教訓なしで終わる。オチは言わない
- **スナップショット型**: 1つのシーンだけ切り取る。背景説明しない
- **投げっぱなし型**: 途中まで語って「まあいいか」で終わる。回収しない
- **自問自答型**: 問いかけ→答えが出ないまま「...」で終わる
- **体験談型**: フック→失敗→気づき→断言。教訓は言語化しない

### 必須要素

以下のうち最低2つを含めること:
- **ビフォーアフターの具体数字** — 「15分→5秒」「月10時間→30分」「0件→47件」のように、変化を数字で見せる。「劇的に」「一気に」はNG。数字で語れないなら書くな
- **泥臭い失敗・葛藤の内心の声** — 「これ自分でやった方が早くない？って何回も思った」「3日かけて作ったDBを全部消した」など、成功の前ではなく失敗そのものにフォーカス
- **超具体的な作業ディテール** — 「金額計算のロジックを直した」「顧客ページに電話番号のフィールドを追加しただけ」など、映像が浮かぶレベルの粒度
- **Notionや武器の固有名詞** — ただし機能名ではなく「何ができるようになったか」で語る

### 感情カーブのバリエーション — 3本同じカーブは不合格

感情の配置を投稿ごとに変える。「冒頭だけ感情、本文は説明」を毎回やるとAI臭い。

- **冒頭感情型**: 1文目で感情爆発 → 本文は事実淡々 → 締めは余韻
- **中盤感情型**: 1文目は事実 → 本文の途中で「いや、これマジで焦った」と感情が入る → 締めは行動
- **終盤感情型**: 1文目は状況描写 → 本文は経緯 → 締めで「...正直泣きそうになった」
- **感情散在型**: 本文の随所に内心の声が入る。「え、待って」「いやこれ無理だろ」「あ、いけるかも」

バッチ生成時は、3本中2本以上が同じ感情カーブパターンにならないこと。

### 引き算の禁止事項

- 先生目線での「便利な機能の紹介・解説」になってしまうこと。
- ふわっとした形容だけで終わること — 「すごい/便利/効率化」
- 固有名詞や極端な数字が入っていないこと。

### 提案方法

- 1つの事象に対して、異なる切り口やトーンで**3〜4パターンのバリエーション**を提示する。複数本を一度に作成する場合はこの限りではない。
- いきなり本文を書き始めず、まず「どんな切り口で行くか」のプランニングを見せてユーザーの合意を得てからドラフトに進む。

## 7. ワークフロー — 5ステップ

### Step 1: 情報収集

- 元となる情報源を読み込む。
- ペルソナ・トンマナを確認する。安易な語尾の付与は避ける。
- 既存のX投稿内容DBを確認し、切り口の重複を避ける。
- コアメッセージを1〜2行で定義する。
- **3つのNotion DBを必ずクエリする** — セクション4参照。
- **ロールモデル確認**: ユーザーが指定した参照アカウントの文体特徴を確認する。確認できない場合は「断片的・投げっぱなし・教訓なし」をデフォルトとする。

### Step 2: ターゲット設定とプランニング

**ターゲット設定が最優先。ターゲットなしにフックは作れない。**

1. **各ポストのターゲットをN=1で設定する**
   - 「1段目の初心者」のような雑なセグメントではなく、具体的な1人を描く
   - その人の状況、悩み、内心の声、行動パターンまで言語化する
   - 例: 「LPを自分で作ろうとしてるフリーランス。外注は高い、自分でやると2週間。デザインもコピーも手探りで完成が見えない」
   - 例: 「ChatGPT使ってるけど出力がイマイチな人。プロンプトのテクニックばかり追ってて、渡す情報が薄いことに気づいてない」

2. **ターゲットの内心の声をフックにする**
   - そのN=1が思わずスクロールを止める1文目を作る
   - ターゲットが「あ、これ自分のことだ」と反射的に思う言葉を叩きつける
   - 弱いフック: 「今日なに進んだ？って夜に振り返って」→ 抽象的で誰にも刺さらない
   - 強いフック: 「LP1枚作るのに何日も溶かしてる。デザインもコピーも手探りで終わりが見えない。」→ LP作成で苦戦してる人がドキッとする

3. **投稿時間帯をターゲットの生活リズムに合わせる**
   - そのターゲットがXを見る時間帯に投稿する
   - フリーランス向け: 朝の作業前、昼休み、夜の振り返り時間
   - 会社員向け: 通勤時間、昼休み、帰宅後
   - 漫然と等間隔に並べない。ターゲットごとに時間帯を判断する

4. **構成・切り口を決める**
   - どのフック種類・投稿タイプで行くかの方針を立てる
   - **複数本作成時は文脈・ストーリー連続性を設計する** — 1日・1週間の中でつながりを持たせる（セクション3参照）
   - 画像ありの投稿を想定し、必要なスクリーンショットを明確にしてユーザーに依頼する

5. **ユーザーに計画を見せて合意を得てからドラフトに進む**

### Step 3: ドラフト作成

- セクション1-5のすべてのルールに従って作成する。
- 3〜4パターンのバリエーションを提示する。

### Step 4: セルフレビュー（ユーザー提示前に必ず実行）

ドラフト完成後、ユーザーに見せる**前に**以下の3軸でセルフレビューを行い、結果を添えて提示する。

**軸1: AIっぽくないか**
- セクション1の禁止パターンに1つでも該当していないか
- 構成が綺麗すぎないか、全投稿が同じ軌道になっていないか
- 過去形3連打→でも転換、格言締め、教科書的接続詞が混入していないか
- 5本並べて「これ全部同じ人が書いたように見えるか？」を自問する

**軸2: ターゲットに刺さるか**
- 1文目を読んだとき、設定したN=1が「あ、自分のことだ」と思えるか
- ターゲット以外の人には刺さらなくてもいい。N=1に全振りできているか
- フックの感情が本文でも生きているか（冒頭だけ感情、本文は説明モードになっていないか）

**軸3: 具体性があるか**
- 数字・固有名詞・映像が浮かぶディテールが入っているか
- 「劇的に変わった」「便利になった」のようなふわっとした表現がないか
- 事実に基づいているか。DBの一次情報から裏取りできない内容を捏造していないか

レビュー結果に問題があれば、ユーザーに見せる前に自分で修正する。
修正から得られた気付きはこのスキルファイルに追記する。

### Step 5: Notion登録

- セクション6のルールに従ってSparkie Posts DBに登録する。
- 登録完了をユーザーに報告する。

### Step 6: AI作業ログへの記録

- 登録完了後、**AI作業ログDB** — `337bd381-95d2-41d5-ada5-3a69c1b64125` — に作業記録を1件作成する。
- 日報DBには記録しない。日報はユーザー自身が書くもの。
- 設定するプロパティ:

| プロパティ | 設定内容 |
|---|---|
| 作業名 | 作業の概要。例: Xポスト5本作成・Notion登録 |
| 日時 | 作業実施日 |
| 作業タイプ | 実行 |
| 担当AI | Cursor |
| ステータス | 完了 |
| 重要度 | 内容に応じて判断 |
| 作業内容 | 何をしたかの要約。情報源・設計方針を含む |
| 作業結果 | 登録件数・予約時間帯・切り口の内訳など具体的な成果 |
| 作業サマリ | 各ポストの1文目フック一覧など、一覧性のある要約 |
| ネクストアクション | 投稿後の確認事項など |

## 9. バッチ生成モード

「明日の投稿3本作って」「3日分の投稿を生成して」「来週分の投稿作って」等のリクエスト時に発動する。
セクション0〜8の全ルールを適用したうえで、以下の戦略に従って自動的にカテゴリ選定・生成・Notion登録を行う。

### 9-1. カテゴリ戦略

投稿カテゴリは5種類。時間帯ごとに適切なカテゴリを重み付きで選定する。

| カテゴリ | 重み | 対象時間帯 |
|---|---|---|
| ノウハウ・Tips | 30% | 08:00, 19:00 |
| 実績・証拠 | 20% | 12:30, 19:00 |
| 煽り・問題提起 | 20% | 12:30, 19:00 |
| ツール紹介 | 15% | 08:00, 19:00 |
| マインドセット | 15% | 08:00, 19:00 |

- 1日3投稿の場合、同じカテゴリが2回以上被らないようにする
- 重みは目安。連日同じカテゴリに偏らないよう意識する

### 9-2. 時間帯ルール

1日あたり3投稿を以下の時間帯に配分する:
- **08:00** — 朝の学び系。ノウハウ・Tips / ツール紹介 / マインドセット
- **12:30** — 昼の刺激系。実績・証拠 / 煽り・問題提起
- **19:00** — 夜のフリー枠。全カテゴリ対象

予約日時は `YYYY-MM-DDTHH:MM:00+09:00` 形式でNotion登録する。

### 9-3. フェーズ管理（30日戦略）

Day番号に応じて投稿のトーンと方向性を切り替える。ユーザーがDay番号を指定しない場合は確認する。

| フェーズ | Day | 方向性 |
|---|---|---|
| 価値提供期 | 1〜7 | フォロワーに役立つ情報を全力で出す。信頼獲得が最優先 |
| 権威性構築期 | 8〜14 | 実績や数字を見せて信頼を積む。具体的な成果を語る |
| 期待感醸成期 | 15〜21 | 次のコンテンツへの期待を高める。伏線を張る |
| ローンチ期 | 22〜30 | 行動喚起を強めて成果に繋げる。CTAは最小限で自然に |

### 9-4. バッチ生成のワークフロー

1. **パラメータ確認**: 対象日付、Day番号、生成日数をユーザーに確認
2. **カテゴリ自動選定**: 時間帯×重みで各投稿のカテゴリを決定。1日内で被りを避ける
3. **一括生成**: フェーズに応じたコンテキストを付与して、セクション0〜5のルールで生成
4. **バッチ内差分チェック**: 全投稿を並べて以下をクロスチェック。2項目以上で被りがあれば書き直し

| チェック項目 | 不合格基準 |
|---|---|
| 結論のキーワード | 3本中2本以上が同じ単語で着地している |
| 締め方の型 | 3本中2本以上が同じ型（格言、行動提案、事実断片、途切れ等） |
| 感情カーブ | 3本中2本以上が同じパターン（冒頭感情型ばかり等） |
| 使用語彙 | 同じ特徴的な単語が2本以上で使われている（「仕組み」「構造」等） |
| 構成の型 | 既存ルール通り。7本中5本以上が同じ型は不合格 |

5. **ユーザー確認**: 生成した全投稿を一覧で提示。修正があれば対応
5. **Notion一括登録**: セクション6のルールに従い、1件ずつSparkie Posts DBに登録
6. **AI作業ログ記録**: セクション7のルールに従い記録

### 9-5. カテゴリ別の生成指針

**ノウハウ・Tips**: AIエージェント・Notionに関する実用的な情報。ターゲットがすぐ使える具体性。固有名詞や数字を入れる。

**実績・証拠**: ぼくの実績を自然に織り交ぜる。自慢にならず事実ベース。泥臭い過程も含める。

**煽り・問題提起**: ターゲットがドキッとする問題提起。現状維持のリスクや盲点を突く。感情を揺さぶる。

**ツール紹介**: 実際に使った体験ベース。具体的で実用的。

**マインドセット**: ぼく自身の体験として語る。説教ではなく背中を押す。**格言・教訓で締めない**。「こう考えるようになった」ではなく「今朝もこうした」という行動事実で終わる。
